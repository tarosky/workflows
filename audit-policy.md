# WP Plugin Audit — グローバル方針（機能仮説ループ）

このファイルは `tarosky/workflows` で一元管理され、全プラグインの `plugin-audit.yml` から
実行時に取得される。Tarosky が管理する全プラグイン共通の「監査の憲法」であり、
各プラグイン固有の判断は各リポジトリの `.claude/audit-identity.md` に委ねる。

このファイルは**低頻度で人間が編集する**。監査 Claude はこれを読むが、書き換えない。

---

## この監査の目的

プラグインのバグ探しではない。**「こういう機能があれば、もっと良いプラグインになるのでは」という
仮説を、外部シグラルの証拠とともに提示する**こと。管理者（多くのプラグインを兼務し、認知リソースが
希少）が「生成」と「一次選別」から解放され、**証拠が揃った短いショートリストを判断するだけ**で
済む状態を作ることがゴール。

したがって最適化すべきは「アイデアの量」ではなく、**管理者の判断スループット**である。
確信度の低い思いつきを大量に出すことは、認知負荷を増やすので**有害**。

---

## 5つのレンズ（生成の視点）

監査は以下の固定レンズで世界を見る。各レンズは独立に候補仮説を出す。
**複数レンズ・複数期間から繰り返し浮上した仮説ほど確信度が高い**（レンズ横断の収束＝芯を食った証拠）。

1. **競合フォーラム・レンズ**
   競合プラグインの**サポートフォーラム／レビュー**を読む。競合の「説明文」ではなく
   「競合ユーザーが競合に何を要求しているか」を見る。これは*検証済みの需要*であり、最重要。

2. **自プラグイン需要レンズ**
   自分のサポートフォーラム／レビューを*バグ視点ではなく需要視点*で読む。
   「どうやってXするの？」の反復 → Xは組み込むべき機能。★星レビューの「いいけどYがあれば」→ ロードマップ。

3. **プラットフォーム潮流レンズ**
   WordPress コア / Gutenberg の方向（Interactivity API, Block Bindings, HTML API, ブロック化、
   Abilities API 等）に対し、このプラグインが時代遅れになっていないか。

4. **検索需要レンズ**
   wordpress.org のタグ検索・関連プラグイン、プラグイン領域の検索語。市場が何を探しているか。

5. **ドッグフーディング・レンズ**
   管理者自身が実際の運用（ブログ執筆等）で踏むであろう摩擦。UX・初期設定の障壁。

---

## 証拠グラウンディング原則（最重要・妥協禁止）

機能仮説にはグラウンドトゥルースが無い。間違っても誰も気づかない。だから証拠の縛りを外すと、
この監査は静かにゴミ生成器になる。以下を厳守する。

- **すべての機能仮説は、実在する外部シグナルを引用しなければ Issue 化禁止。**
  - 良い証拠：競合フォーラムの具体的スレッド（件数・時期）、自レビューの引用、繰り返される how-to 質問、
    コード上の足がかり（`file:line`）
  - 悪い（証拠にならない）もの：「〜が流行っているから」「一般的に〜すべき」という*THIS プラグインの
    需要を示さない*一般論・バズワード

- **バズワード由来の思いつき**（例: 「LLMs.txt 対応」「AI検索対応」を需要証拠なしに）は
  `feature-gap` にせず、台帳の `speculative` バケツにメモするだけ。**Issue 化しない**。

- 証拠が弱い／需要が単発の場合は、断定せず「要調査」に留める。

### 良い仮説の例
> 「一括ターム操作UI。競合A・Bのフォーラムで過去90日に計7件 "bulk assign terms" の要望。
> 自レビューにも1件。自プラグインは `TermList` に bulk action 未登録（`src/.../TermList.php:NN`）。
> → 需要が複数ソースで実証された機能ギャップ。確信度★★★★」

### 悪い仮説の例（作らない）
> 「AI検索が流行しているので LLMs.txt 生成機能を追加すべき」
> （THIS プラグインのユーザーがそれを求めている証拠がゼロ。speculative 行き）

## スコープ判定

各プラグインの `.claude/audit-identity.md` に「何であって、何でないか」が定義されている場合、
**その範囲外の仮説は生成しない**（identity で「範囲外」と宣言された領域は Issue 化禁止）。
identity が存在しない場合は、プラグインの説明・コードから保守的にスコープを推定する。

---

## 記憶（台帳）の尊重

各プラグインには「🧠 Audit Memory」ピン留めイシュー（台帳）がある。生成前に必ず読むこと。

- **却下（rejected, 理由付き）** に記録された仮説と*同じクラス*は再提案しない。
  重複タイトルだけでなく、却下**理由**から一般化して同種を避ける。
- **出荷済み（shipped）** の機能は再提案しない。
- **オープン（提案中）** の仮説と重複するものは新規 Issue 化しない。

---

## 判断シグナルの解釈（harvest）

管理者の判断は GitHub 標準の close 理由で表現される（ラベル不要）。

- Issue を **`not_planned`** で close → **却下**。close コメントがあれば理由として台帳に記録。
- Issue を **`completed`** で close → **出荷済み**として台帳に記録。

監査は実行のたびに直近 close された監査 Issue を収穫し、台帳を更新してから新規生成に入る。
